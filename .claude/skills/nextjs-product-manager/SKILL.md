# Next.js 기능 기획 스킬

## 1. 기존 프로젝트 파악 (항상 먼저)

명세 작성 전에 현재 상태를 파악한다:
- `app/` 디렉토리: 기존 페이지와 라우트 확인
- `components/`: 재사용 가능한 기존 컴포넌트 목록
- `types/index.ts`: 기존 타입 정의
- `store/`: 기존 상태 관리 구조
- 기존 디자인 시스템 색상/패턴

## 2. 화면 목록 정의

```
| URL | 역할 | 인증 필요 | 레이아웃 |
|-----|------|-----------|----------|
| /   | 대시보드 | Yes | 헤더 + 사이드바 + 메인 |
| /login | 로그인 | No | 전체 화면 |
| /new-page | {역할} | Yes/No | {레이아웃} |
```

새 페이지가 기존 레이아웃(`app/layout.tsx`)을 공유하는지, 독립 레이아웃이 필요한지 명시한다.

## 3. 사용자 흐름 정의

```
[진입점] → [화면A] → [인터랙션] → [화면B] → [완료 상태]

예:
[대시보드] → [+ 지출 추가 클릭] → [모달 오픈] → [폼 입력] → [저장] → [테이블 갱신]
```

에러 흐름도 함께 정의한다:
```
[폼 입력] → [저장 실패] → [에러 메시지 표시] → [재시도 또는 취소]
```

## 4. 화면별 데이터 & 인터랙션 명세

각 화면에 대해:
```
### {화면명} (/path)

**표시 데이터:**
- {데이터1}: {출처 — API 엔드포인트 또는 로컬 상태}
- {데이터2}: ...

**사용자 인터랙션:**
- {버튼/폼}: {동작} → {결과}
- {필터}: {동작} → {결과}

**빈 상태 / 로딩 상태 / 에러 상태:**
- 로딩: {표시 방법}
- 빈 데이터: {표시 방법}
- 에러: {표시 방법}
```

## 5. 백엔드 API 의존성

crow-backend의 기존 API와 매핑한다:

```
| 기능 | HTTP | 엔드포인트 | 인증 |
|------|------|-----------|------|
| 지출 목록 | GET | /expenses/expenses/ | JWT |
| 지출 생성 | POST | /expenses/expenses/ | JWT |
| 예산 요약 | GET | /expenses/expenses/summary/ | JWT |
| 카드 SMS 수신 | POST | /receivers/webhook/ | 없음 |
```

새 API가 필요한 경우 crow-backend에 추가가 필요함을 명시한다.
